DELIVERY ARCHITECTURE
Five fulfilment modes at the consensus layer

A single PURCHASES transaction attachment supports five distinct fulfilment patterns, from atomic instant delivery to third-party logistics with encrypted proofs. The fulfilment mode determines the escrow release conditions, deadline enforcement, and delivery confirmation mechanism. All enforced by the blockchain engine, not by the application.

⚡

Atomic delivery. Payment and delivery are atomic in the same transaction. No escrow period. No deadline. No confirmation required. Used for immediate digital goods e.g. access keys, software licences, on-chain content tokens, or any product where delivery is instantaneous at payment. The delivery can also be used for in-store purchases for Point-of-Sale payments for goods.

BUYER PROTECTION FOR ALL FIVE MODES
In every fulfilment mode except INSTANT, the buyer's payment is held in protocol escrow until delivery is confirmed or the deadline expires. The protocol and not the merchant or an escrow agent holds and releases the funds. There is no way for a merchant to receive payment without either the customer confirming delivery before the agreed-upon block height deadline is reached.

MERCHANT ACCOUNT ENFORCEMENT

MERCHANT Account Type Required

Receipt of PURCHASE transactions is restricted to MERCHANT-type accounts at gate 4 (Account Type Gate) of the consensus pipeline. Non-MERCHANT accounts cannot receive PURCHASE transactions which are rejected before they reache the mempool.

  • Only AccountType.MERCHANT can receive PURCHASE
  • Enforced at consensus with not application-layer check
  • MERCHANT status requires CHAIN_USER permission (KYC)
  • Every merchant is identifiable and KYC-accountable
  • No anonymous merchants structurally possible

STORE_PURCHASE Routing

If a buyer attempts to send a purchase to a non-MERCHANT account, the transaction fails before any funds are moved.

  • Recipient account type validated at consensus
  • Prevents misdirected payments entirely
  • Merchant cannot impersonate another account type
  • Both sender and recipient roles enforced by protocol
03
KYBER ENCRYPTION
Zero plaintext customer data on-chain

Every purchase payload includes customer delivery address, order details, personal data, and 3PL tracking information Kyber-encrypted to the merchant's public key before being stored in the transaction attachment. Only the merchant with the corresponding Kyber-512 private key can decrypt it. Node operators, blockchain explorers, and other participants see only ciphertext.

Recipient Privacy by Default
Customer identity, delivery address, and order details are encrypted using the hybrid Kyber KEM + AES-256-GCM pattern before being included in the PURCHASES transaction. This happens in the wallet software before the transaction is submitted to the network and the plaintext never appears in any node's memory after the sender's device.
3PL Proof Encryption
When a buyer submits a delivery confirmation after successful delivery by either the merchant or a third-party logistics provider, the proof payload is also Kyber-encrypted to the merchant's public key. Only the merchant can decrypt and read the delivery confirmation proof — the 3PL and node operators see only the ciphertext and the on-chain proof hash.
Quantum-Resistant by Default
ML-KEM-512 (NIST FIPS 203) ensures that purchase payloads captured and archived today cannot be decrypted by future quantum computers. Customer order histories, addresses, and personal data stored in every PURCHASES transaction are structurally protected against harvest-now-decrypt-later attacks.
On-Chain Privacy Anchoring
The encrypted data hash and ciphertext is permanently stored in the blockchain state, providing an immutable, cryptographically-anchored record of every transaction payload. The merchant holds the sensitive information; the blockchain holds the commitment. This enables dispute resolution without exposing sensitive data to the network.
HYBRID ENCRYPTION PATTERN

ML-KEM-512 + AES-256

Purchase payloads use the standard post-quantum hybrid encryption pattern: (1) Kyber-512 KEM encapsulates a 32-byte shared secret using the recipient's 800-byte Kyber public key, (2) AES-256 encrypts the payload using that shared secret, (3) the 768-byte Kyber ciphertext and AES-GCM ciphertext are concatenated into the transaction attachment.

  • Kyber encapsulation: 768 bytes in transaction attachment
  • AES authenticated encryption
  • Only recipient Kyber private key can decapsulate
  • AES tag ensures payload integrity with tampering detectable
  • Hardware-accelerated AES-NI for bulk encryption performance
  • Zero plaintext exposed at any point in the protocol pipeline

What the Protocol Stores vs What It Enforces

The on-chain record is split: public fields are stored in plaintext (for routing and enforcement), sensitive fields are stored as Kyber-encrypted ciphertext.

  • Public (plaintext): merchant account ID, purchase amount, currency, deadline height, reward config
  • Encrypted (ciphertext): delivery address, order items, personal data, contact info
  • Protocol enforces: escrow, deadline, reward, merchant type — all from public fields only
  • Privacy: all sensitive customer data encrypted before network submission

ESCROW & BUYER PROTECTION
Protocol-enforced escrow without a third party

Funds are locked in the blockchain's own escrow state at purchase. Release conditions are protocol-enforced requiring no escrow agent, no dispute resolution service and no trust in the merchant. The blockchain is the escrow. There are only three outcomes: buyer confirms delivery and merchant receives payment; deadline expires and buyer receives automatic refund.

ESCROW LIFECYCLE
PURCHASES submitted
Buyer submits PURCHASES transaction. Protocol atomically: (1) debits buyer balance, (2) holds funds in-transit, (3) creates PurchaseEntity in blockchain state with fulfilmentMode, deadlineHeight, and encrypted payload.
Merchant receives notification
Merchant's wallet notifies of new purchase. The encrypted payload is readable only by the merchant. Protocol state shows PendingDelivery status which is public and verifiable by both parties.
getAccount?requestType=getPurchases
Merchant ships / delivers
For PHYSICAL mode: merchant ships the order off-chain. For DIGITAL: merchant uploads Kyber-encrypted payload on-chain. For 3PL: third-party logistics provider handles delivery.
DELIVERY_CONFIRMATION or encrypted payload
Delivery confirmed and funds are released
Buyer submits DELIVERY_CONFIRMATION. Protocol atomically: (1) releases funds to merchant, (2) merchant distributes loyalty rewards to buyer, (3) marks PurchaseEntity as Delivered.
Deadline expires — auto-refund
If DELIVERY_CONFIRMATION is not submitted before deliveryDeadlineHeight, AFTER_BLOCK_APPLY atomically: (1) refunds funds to buyer, (2) marks PurchaseEntity as EXPIRED, (3) recorded for public verification.

What You No Longer Need

The protocol replaces every third-party intermediary in the commerce payment stack.

  • Payment processor (Stripe, PayPal, M-Pesa integration)
  • Escrow service or trust intermediary
  • Dispute resolution platform (chargebacks, claims)
  • Fraud monitoring (merchant type verified at genesis)
  • Customer address database (encrypted on-chain)
  • Tax calculation middleware (protocol withholds VAT/sales tax)
  • Delivery confirmation tracking for payment release
  • Loyalty programme platform and reward redemption API

What the Protocol Guarantees

Every buyer protection listed below is enforced by the consensus engine and not by application code, operator policy, or contractual obligation.

  • Funds cannot reach merchant before delivery confirmation
  • Buyer receives full refund if deadline expires automatically
  • Merchant cannot cancel a purchase and keep payment
  • Loyalty rewards can be distributed at confirmation.
  • VAT and sales tax withheld automatically and credited to tax collector atomically and not by the merchant
  • All purchase events permanently recorded for immutable evidence

LOYALTY REWARDS
Multi-asset rewards distributed at confirmation

Merchants configure loyalty reward amounts in any asset type i.e chain coin, ShamwariPay currency, or Totem digital asset. Rewards are distributed at DELIVERY_CONFIRMATION in the same atomic block or after for long running promotions. No separate loyalty programme, no points platform, no redemption infrastructure required.

🪙
Coin Reward
CHAIN NATIVE COIN
Distributethe native chain coin as loyalty rewards. Accumulated coin can be used for chain fees.
💵
Currency Reward
ZWG / ZAR / CAPITAL
Distribute jurisdiction-specific currency (ZWG, ZAR) or CAPITAL as rewards. Directly spendable on the same chain for future purchases, insurance premiums, subscription billing, or DEX trading. Highest utility for emerging market consumers with limited access to other asset types.
🔷
Totem Reward
ANY APPROVED TOTEM ASSET
Distribute branded loyalty tokens, collectibles, or fractional ownership tokens as rewards. Full TotemApproval compliance applies i.e reward tokens must be APPROVED status on the chain. Holders accumulate positions eligible for dividend distributions. Suitable for brand loyalty programmes with genuine asset value.

Reward Distribution

Reward parameters are set after a purchase has been fulfilled or atomically with delivery.

  • rewardType: COIN | CURRENCY | TOTEM
  • rewardAmount: fixed amount for the purchase
  • rewardAssetId: required for CURRENCY and TOTEM types
  • Rewards are determined by the merchant
  • Setting rewardAmount = 0 negates rewards for that listing
  • Different purchases can have different reward amounts or types

Atomic Distribution

Reward distribution can be a separate transaction or a delayed batch. It can execute atomically in the same block as DELIVERY_CONFIRMATION or DELIVERY as part of the same database transaction.

  • TotemApproval status checked at distribution so that only APPROVED assets can be used as rewards
  • Tax withholding applied to reward if applicable tax type configured
  • CurrencyTaxRecord created per reward if taxable
  • All reward events permanently recorded in purchase history

MERCHANT REPUTATION
Immutable on-chain purchase and delivery data

As a result of the public purchase and delivery records, a merchant's reliability can be established by looking up their purchase completion thereby creating a transparent reputation ranking with no manipulation prone rating and review systems.

Protocol-Automatic Penalty
Everytime a purchase deadline expires without delivery confirmation, the merchant's delivery rate is publicly reduced. No buyer dispute required. No platform review process. The missed delivery is permanently recorded as an event in the merchant's on-chain reputation history.
Publicly Verifiable Delivery Record
Any participant, buyer, regulator, marketplace operator, or competitor, can query any merchant account's full transaction history and refund history via the REST API in real-time. The blockchain is the ultimate reputation record.
Immutable History
Once a purchase is closed, whether automatic (missed deadline) or delivery, it cannot be removed. Merchants cannot delete either completed or expired purchase records. There is no support ticket that reverses a protocol-recorded delivery failure. The record is permanent.

AFTER_BLOCK_APPLY — Automatic Refund

The PURCHASES escrow automatic refund is not triggered by a transaction. It is triggered by the AFTER_BLOCK_APPLY event when the block height matches deliveryDeadlineHeight.

  • No buyer action required for expired delivery refund
  • Protocol checks all PendingDelivery purchases every block
  • Refund executes in same block as deadline height
  • Merchant reputation decremented in same atomic operation
  • Zero gas cost to buyer for receiving refund

Escrow Account Architecture

Escrow is held in a dedicated protocol-managed escrow account — not in the merchant's balance. This eliminates merchant insolvency risk for pending purchases.

  • Escrow held at protocol level — not merchant balance
  • Merchant cannot access escrowed funds before confirmation
  • Buyer insolvency during escrow: original debit holds
  • No counterparty credit risk during delivery window
  • Multiple concurrent purchases escrow independently

COMPARISON
Protocol commerce vs traditional infrastructure

The fundamental difference: in Shamwari's commerce system, the protocol performs every role that currently requires a third-party service provider. This eliminates integration complexity, counterparty risk, and data privacy vulnerabilities.

SHAMWARI PROTOCOL COMMERCE
  • Payment processing by consensus — no processor dependency or uptime risk
  • Escrow by protocol — no escrow agent, no trust relationship required
  • Buyer protection enforced by blockchain — automatic, cannot be overridden
  • Tax withholding automatic — VAT, sales tax calculated and withheld at execution
  • Customer data Kyber-encrypted — no plaintext in blockchain state
  • Merchant reputation maintained by protocol — immutable, no platform censorship
  • Loyalty rewards distributed at confirmation — no separate programme infrastructure
  • 3PL delivery proofs encrypted on-chain — logistics verifiable and private
TRADITIONAL COMMERCE STACK
  • Payment processor required (Stripe/PayPal/M-Pesa) — 2–4% fee per transaction
  • Escrow requires third-party service or contractual trust in merchant
  • Buyer protection via chargeback — days/weeks process, bank intermediary
  • Tax calculation middleware (TaxJar, Avalara) — additional integration and cost
  • Customer PII in plaintext in payment processor database — GDPR/data breach risk
  • Reputation systems controlled by platform — shadow-banning, censorship possible
  • Loyalty programme requires separate vendor (Loyalty+ etc.) — API integration
  • Logistics verification requires webhook integration with courier APIs
